What Went Well

  • The pivot was the right call. v1 would have added 400+ lines of new JS encoder code. v2 deleted 413 lines and added 14. Net: −408 lines. Simpler, safer, less to maintain.
  • Existing infrastructure did the heavy lifting. The prime already generated qr_data. The relay already extracted it. The browser already had renderQR(). The gap was just plumbing — 3 small patches.
  • Relay SSH worked. gcloud compute ssh with the bundled Python resolved the path issue. Compile → deploy took one cycle.
  • E2E test was thorough. 10 assertions including finder pattern verification and duplicate rejection. Caught the initial Cloudflare propagation delay (first request failed, second passed).

What Could Be Better

  • gcloud Python path is fragile. The bundled Python at google-cloud-sdk/platform/bundledpython/python.exe needs CLOUDSDK_PYTHON set every time. Should add this to shell profile or a wrapper script.
  • First E2E test hit before Cloudflare propagated. Should add a brief wait or retry after wrangler pages deploy before testing the live URL.
  • Relay compile flags not documented. Had to discover -lresolv and include paths by trial. Should save the build command somewhere on the VM.
  • v1 plan was wasted work. Should have asked about the all-C approach earlier in the ~I phase. The inspection found the C version works — the obvious next question was “why not just use it?”

Commitments

  • Manual QR scan still needed. E2E verifies the data pipeline. A human needs to scan the QR with a phone to confirm it works end-to-end in the real browser UI.
  • Document relay build command. Save gcc flags + include paths in a Makefile or script on the VM.
