Platform Shell

Multi-tenant + Provisioning + Dynamic Branding + 3 Dashboard Layouts
Zero application features. Platform only.

Intent

Build the platform that hosts tenants, not the app that serves users.

When this plan ships, any institution can be provisioned with its own branding, its own login, its own dashboard — but the actual application features (queue, directory, profiles, enrichment, outreach) are not wired up yet. Those come later, on top of this shell.

What this is NOT

  • No alumni data layer. No CSV import. No enrichment.
  • No queue, directory, or profile views wired to data.
  • No cause-effect engine. No triggers.
  • No Firestore. No Workers API beyond tenant config.
  • The app views exist as empty layout shells with placeholder content.

Key Design Decision

Tenant config lives in the C org store (relay/prime), not in Cloudflare KV or D1. The Worker fetches tenant config via /api/wire on first request, caches it in memory for the session. Single source of truth = triples in C.

For the platform shell phase, we can bootstrap with a static tenant config JSON served from the repo while the C endpoint is wired up. The browser reads /api/tenant/config → Worker returns cached config or falls back to static JSON.

Branding System

How It Works

  1. Browser loads — detects tenant from subdomain or path
  2. Fetches config — GET /api/tenant/config → JSON with brand colors, name, logo, campuses
  3. Injects CSS vars — overrides :root custom properties with tenant colors
  4. Renders chrome — nav shows tenant name/logo, login shows tenant brand, dashboard uses tenant campuses

CSS Variable Injection

// brand.js — runs before any view renders
async function loadTenantBrand() {
  const config = await fetch('/api/tenant/config').then(r => r.json());
  const root = document.documentElement.style;

  // Override CSS custom properties
  root.setProperty('--primary',      config.brand.primary);
  root.setProperty('--primary-dark', config.brand.primaryDark);
  root.setProperty('--accent',       config.brand.accent);
  root.setProperty('--accent-light', config.brand.accentLight);
  root.setProperty('--background',   config.brand.background);

  // Inject text
  document.querySelector('.nav-brand').textContent = config.shortName;
  document.querySelector('.nav-logo').src = config.brand.logoUrl;

  return config;
}

Tenant Routing

How Requests Find Their Tenant

Request: https://westernu.lakemedical.org/login.html
                 ^^^^^^^^
                 subdomain = tenant_id

Worker reads Host header → extracts subdomain → tenant_id = "westernu"
All subsequent API calls include tenant_id:
  /api/tenant/config?t=westernu
  /api/wire?t=westernu (auth scoped to tenant)

Fallback for dev/single-tenant:
  ?tenant=westernu query param
  localStorage['tenant_id'] override

Session Scoping

Session tokens include a tenant_id claim. If a session token from westernu is used against anotheruniv, the relay returns 403. Zero cross-tenant session leakage.

  • Session scoping: tenant_id in auth requests, relay validates tenant claim

  • Queue view → empty state: "No triggers yet. Import alumni data to get started."

  • Dashboard → empty stat cards + "No data" chart areas
  • Keep: nav structure, sidebar, session timer, campus toggle (dynamic from config)

After this: the app is a branded, authenticated, multi-tenant shell. A provisioned tenant logs in and sees empty views with clear CTAs for what comes next (import, enrichment).

Constraints

Rules

  • Zero frameworks. Vanilla HTML/CSS/JS only. No React, no Tailwind, no bundlers.
  • No application features. If it needs alumni data to work, it's out of scope. Empty states only.
  • CSS variables for everything. No hardcoded brand colors anywhere after Phase 1.
  • Tenant config is the single source. Every piece of institution-specific text comes from config, never from code.
  • Static bootstrap first. tenant-config.json serves as the config source until the C relay endpoint is wired. No blocking on C work.
  • 3 dashboards must be distinct pages/layouts. Not tabs on one page. Different URL, different nav, different role.
