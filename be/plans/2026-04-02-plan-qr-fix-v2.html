~P: Fix QR Code v2 — All C

Server-side QR generation · Delete qr.js · Zero new JavaScript · lakemedical.org

Plan — Awaiting Approval

Intent

Why: Users cannot register on lakemedical.org because the browser’s JS QR encoder is broken (4 fatal bugs). Meanwhile, the prime already generates correct QR data in C (qr.c, 606 lines, 200+ test assertions).

Purpose: Pipe the prime’s server-generated QR matrix through relay → Worker → browser. No new JS encoder. Delete the broken one.

What: The browser receives qr_data (base64 QR matrix) + qr_size from the server and renders it on canvas. All QR generation stays in C.

Why All C

The prime’s catalog_server.c already calls qr_encode() during registration and returns 5 triples:

resp[0] = {resp, status,   ok}
resp[1] = {resp, totp_uri, otpauth://totp/NOUS:user?secret=...&issuer=NOUS}
resp[2] = {resp, qr_size,  37}
resp[3] = {resp, qr_data,  BASE64_ENCODED_MATRIX}
resp[4] = {resp, role,     user}

The qr_data is a base64-encoded byte array of qr_size × qr_size bytes, each 0x00 (white) or 0x01 (dark). This is the proven, tested QR output from qr.c.

The data is already being generated. It just needs to flow through to the browser. Writing a new JS QR encoder would be duplicating 606 lines of proven C for no reason.

Approach

Phase 1: Relay — Add qr_data to /register JSON

The relay’s handle_register_http() already extracts otpauth and qr_size from the prime’s wire response. Add one more extraction:

/* Already done: */
char *otpauth = wire_find2(reg_resp, n, "resp", "totp_uri");
char *qr_size = wire_find2(reg_resp, n, "resp", "qr_size");

/* Add: */
char *qr_data = wire_find2(reg_resp, n, "resp", "qr_data");

/* In JSON response, add: */
"qr_data": "%s"    /* base64 string from prime */

Compile, deploy to relay VM. ~5 lines.

Phase 2: Worker — Pass qr_data + qr_size as triples

In handleRegister(), after the existing json.otpauth triple:

/* Already done: */
if (json.otpauth) triples.push({ s: json.otpauth, p: 'signal', o: 'none' });

/* Add: */
if (json.qr_data) triples.push({ s: json.qr_data, p: 'qr_data', o: 'none' });
if (json.qr_size) triples.push({ s: String(json.qr_size), p: 'qr_size', o: 'none' });

Deploy via npx wrangler pages deploy. ~4 lines.

Phase 3: Auth Gate — Render server QR

In the register success handler, replace the qr_encode() call:

/* OLD (broken): */
var otpUri = nous_find(triples, 'signal');
if (otpUri && typeof qr_encode === 'function') {
  var qr = qr_encode(otpUri);
  if (qr) renderQR(qrCanvas, qr.data, qr.size);
}

/* NEW (server QR): */
var qrB64  = nous_find(triples, 'qr_data');
var qrSize = parseInt(nous_find(triples, 'qr_size'), 10);
if (qrB64 && qrSize) {
  var raw = atob(qrB64);
  var matrix = new Uint8Array(raw.length);
  for (var i = 0; i < raw.length; i++) matrix[i] = raw.charCodeAt(i);
  renderQR(qrCanvas, matrix, qrSize);
}

The existing renderQR() function already takes (canvas, matrix, size) — no changes needed.

Phase 4: Delete qr.js

  • Delete alumni/js/utils/qr.js (413 lines of broken code)
  • Remove <script src="./js/utils/qr.js"></script> from alumni/index.html

Constraints

  • All QR generation in C — zero new JS encoders
  • No new JS files — only modify existing ones
  • Vanilla JS only — no frameworks, no build tools
  • Base64 decode uses atob() — built into every browser
  • The qr_data base64 string for a V5 QR (37×37) is ~1.8 KB — negligible wire cost
  • Relay patch requires SSH to relay VM + recompile
