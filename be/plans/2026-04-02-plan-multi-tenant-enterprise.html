Plan: Multi-Tenant Alumni Engagement Platform

Enterprise grade · Pure C core · Roll your own · kASTIL/nous foundation

PLAN — AWAITING APPROVAL

Intent

Transform the single-tenant lakemedical.org alumni prototype into an enterprise-grade, multi-tenant platform where any college or university can provision their own branded alumni engagement instance — with all core logic in pure C, reusing the kASTIL/nous library stack.

  • Features are universal — same engine for every tenant
  • Branding is per-tenant — colors, logo, name, domain
  • Data is isolated — each tenant has its own encrypted triple store
  • Enrichment runs server-side in C — not in JS Workers

Multi-Tenancy Model

Tenant Isolation

/var/nous/tenants/
  lakemedical/
    config.tri        # tenant config triples (name, colors, domain, logo)
    alumni.store/     # encrypted triple store (store.h)
    users/            # TOTP secrets, user profiles
    secrets.vault     # sealed tenant PSK + API keys (vault.h)
    refresh.cursor    # enrichment batch position
  westernu-comp/
    config.tri
    alumni.store/
    users/
    secrets.vault
    refresh.cursor
  ...

Tenant Routing

  • Domain-based: Request hits Cloudflare → Worker reads Host header → passes tenant_id to alumni-server
  • PSK derivation: tenant_psk = HMAC-SHA-256(master_key, tenant_id) — no per-tenant secret management needed
  • Session scope: Token includes tenant claim — {token, meta, tenant:lakemedical}
  • Data path: /var/nous/tenants/{tenant_id}/alumni.store/

Tenant Provisioning (C)

alumni_server --provision <tenant_id> \
  --name "Lake Medical Alumni" \
  --domain "lakemedical.org" \
  --color-primary "#3779B5" \
  --color-accent "#90C4CF" \
  --logo-url "/assets/logo.svg"

  1. mkdir /var/nous/tenants/{tenant_id}/
  2. triple_store_open_encrypted() with derived key
  3. Insert config triples (name, domain, colors, logo)
  4. vault_seal() tenant secrets
  5. Ready to accept registrations

The alumni-server Binary (Pure C)

New C binary — the multi-tenant core

One process. HTTP listener on port 9080. Handles all tenant data, enrichment, auth delegation, and config serving.

NPI Enrichment Workflow (Pure C)

Scheduler fires NPI job for tenant
  │
  ├─ 1. Read refresh.cursor → start position
  ├─ 2. store.h: query alumni with status=pending-enrichment
  │     OR alumni with enriched-at older than 30 days
  │
  ├─ 3. For each alumnus (batch of 200):
  │     │
  │     ├─ http.h: GET https://npiregistry.cms.hhs.gov/api/
  │     │    ?first_name={first}&last_name={last}
  │     │    &enumeration_type=NPI-1&limit=20
  │     │
  │     ├─ read.h: parse JSON response
  │     │
  │     ├─ Score matches (same algorithm as enrich.mjs):
  │     │    +10  credential contains "DO"
  │     │    +3   enumeration year within ±4 of class_year
  │     │    +2   first name exact match
  │     │    Accept if score ≥ 5
  │     │
  │     ├─ If matched:
  │     │    store.h: insert/update triples:
  │     │      alumni:{id}  has-npi        "1234567890"
  │     │      alumni:{id}  specialty      "family-medicine"
  │     │      alumni:{id}  practice-city  "Orange"
  │     │      alumni:{id}  practice-state "CA"
  │     │      alumni:{id}  license-status "active"
  │     │      alumni:{id}  enriched-at    "2026-04-02T02:15:00Z"
  │     │      alumni:{id}  status         "enriched"
  │     │
  │     ├─ Detect changes (diff previous triples):
  │     │    If practice-city changed → create practice-move trigger
  │     │    If specialty changed → create credential-change trigger
  │     │
  │     └─ sleep(200ms) — NPI rate limit
  │
  ├─ 4. Save refresh.cursor
  └─ 5. journal.h: commit

PubMed Enrichment Workflow (Pure C)

Scheduler fires PubMed job for tenant (weekly)
  │
  ├─ 1. For each alumnus (batch of 100):
  │     │
  │     ├─ http.h: GET eutils/esearch.fcgi
  │     │    ?db=pubmed&term={last}+{first_initial}[Author]
  │     │    &retmax=5&sort=pub_date
  │     │
  │     ├─ read.h: parse XML → extract PMIDs
  │     │
  │     ├─ http.h: GET eutils/esummary.fcgi
  │     │    ?db=pubmed&id={pmid1,pmid2,...}
  │     │
  │     ├─ read.h: parse XML → title, journal, date, authors
  │     │
  │     ├─ For each new publication (not in store):
  │     │    store.h: insert triples:
  │     │      alumni:{id}  publication    "pmid:38291045"
  │     │      pub:38291045 title          "Clinical outcomes..."
  │     │      pub:38291045 journal        "JAMA"
  │     │      pub:38291045 pub-date       "2026-01-15"
  │     │      alumni:{id}  tag            "published-researcher"
  │     │
  │     │    Create publication trigger:
  │     │      trigger:{t}  type           "publication"
  │     │      trigger:{t}  alumni         "alumni:{id}"
  │     │      trigger:{t}  detail         "New paper in JAMA"
  │     │
  │     └─ sleep(500ms) — PubMed rate limit
  │
  └─ 2. journal.h: commit

What Stays JS (Platform Constraints)

  • Cloudflare Workers — must be JS/WASM. Cloudflare doesn't run native C. Workers become thin HTTP proxies: receive browser request, forward to alumni-server, return response. No business logic in Workers.
  • Browser — must be JS/HTML/CSS. Vanilla only. No frameworks. The browser is a renderer, not a processor.

Everything else is C. If it runs on our metal, it's C.

Constraints

  • Pure C for all server-side logic. JS only where platform mandates it (Cloudflare Workers, browser).
  • No frameworks. No ORM, no Express, no React. Vanilla everything.
  • No external dependencies. All crypto, HTTP, storage, parsing from kASTIL/nous.
  • FERPA compliant. Only public federal APIs (NPI, PubMed) + data the institution provides. No scraping.
  • Encryption at rest. Every tenant store uses AES-256-GCM via store.h.
  • Cross-platform build. Compiles on Linux (production) and Windows (development).
  • NPI rate limit: max 5 req/sec. PubMed: max 3 req/sec without API key.
