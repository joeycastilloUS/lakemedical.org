Epic 1 — Authentication & Sessions

7 stories
PB-001P1

As a coordinator, I want to register with a username and TOTP authenticator app, so that I can access the platform without passwords.

AC: Enter username → server generates TOTP secret → QR code displayed → scan with authenticator → enter 6-digit code → account created.

PB-002P1

As a coordinator, I want to log in with my username and a 6-digit TOTP code, so that I can resume my work securely.

AC: Username + code → validated via wire protocol → session minted → redirected to app. Invalid code shows error.

PB-003P1

As a coordinator, I want my session to automatically expire after 15 minutes of inactivity, so that unattended screens don't expose data.

AC: 900s TTL. Timer visible. Warning at 2 minutes (yellow). Critical at 1 minute (red). Auto-redirect to login on expiry.

PB-004P1

As a coordinator, I want the app to check my session on page load, so that I'm redirected to login if my session is invalid.

AC: On load, verify session token. Valid → show app. Invalid/expired → redirect to login.html. No flash of app content.

PB-005P1

As a coordinator, I want to see a QR code during registration that I can scan with Google Authenticator, so that I can set up TOTP without typing a secret key.

AC: QR code rendered from server-generated data. Scannable by Google Authenticator, Authy, 1Password. otpauth:// URI with correct issuer + account.

PB-006P2

As an admin, I want sessions bound to the originating IP, so that stolen tokens can't be used from another location.

AC: Session token includes IP claim. Requests from different IP → session rejected → re-auth required.

PB-007P2

As an admin, I want all auth traffic encrypted via binary wire protocol, so that credentials never travel as plaintext JSON.

AC: NTRP0001 binary frames. AES-256-GCM 0xCA envelope between Worker and relay. No credentials in query params or plain headers.

Epic 2 — Queue & Outreach Triggers

10 stories
PB-008P1

As a coordinator, I want to see today's queue of outreach triggers sorted by priority, so that I work the most important items first.

AC: Queue view is default landing. Shows date, greeting, item count. Cards sorted: high → medium → low, then by newest date. Each card shows: name, specialty, class year, trigger type, priority badge, summary.

PB-009P1

As a coordinator, I want each trigger card to show contextual insights (campus proximity, student matches, days since last contact), so that I can prioritize with real context, not just a label.

AC: Up to 4 insight bullets per card. Computed from alumni data: distance to campus, students wanting that specialty, last touchpoint age, days overdue.

PB-010P1

As a coordinator, I want to filter the queue by trigger type (practice moves, publications, follow-ups, etc.), so that I can focus on one category at a time.

AC: Toggle pills for: All, Practice Moves, Credentials, Publications, Follow-ups, Milestones, Re-engagement, SOAP, Achievements. Active pill highlighted. Count updates.

PB-011P1

As a coordinator, I want to filter the queue by campus (All / Pomona / Lebanon), so that I can work one campus at a time.

AC: Campus toggle. Filters trigger cards by campus field. "All" shows both. Count updates per filter.

PB-012P1

As a coordinator, I want to dismiss a trigger from my queue, so that I can clear items I've handled or don't need to act on.

AC: [Dismiss] button on each card. Card removed from queue. Moved to archived section. Queue count decrements.

PB-013P1

As a coordinator, I want to restore a dismissed trigger, so that I can undo an accidental dismiss.

AC: Archived section (collapsible) shows dismissed items. [Restore] button returns item to active queue. Archived section hidden when empty.

PB-014P2

As a coordinator, I want trigger cards color-coded by type (blue=moves, gold=credentials, green=publications, red=follow-ups), so that I can visually scan the queue at a glance.

AC: 4px left border color per trigger type. Icon + label matches type. 8 distinct trigger type colors.

PB-015P2

As a coordinator, I want a SOAP Mentorship card in my queue during Match Day season, so that I can connect unmatched students with alumni mentors.

AC: Gold-bordered special card. Shows unmatched student count, mentor names, inspirational quote. Links to mentor profiles.

PB-016P2

As a coordinator, I want to click "View Profile" on any trigger card, so that I can see the full alumni record before taking action.

AC: Click navigates to profile view. Back button returns to queue (not directory). Trigger context preserved.

PB-017P3

As a coordinator, I want the queue badge count visible in the navigation at all times, so that I know when new triggers arrive without switching views.

AC: Badge on Queue nav item shows pending count. Updates when items dismissed/restored. Visible from directory, profile, and dashboard views.

Epic 7 — Data Import & Enrichment

8 stories
PB-051P1

As an admin, I want to upload a CSV of graduates (first_name, last_name, class_year, campus), so that alumni records are created in the system from the registrar's data.

AC: CSV upload form. Accepts: first_name, last_name, class_year, campus (required) + email, student_id (optional). Validates format. Shows import progress. Returns: imported count, duplicates skipped, errors.

PB-052P1

As an admin, I want imported alumni automatically enriched with NPI Registry data (NPI number, specialty, practice location, license status), so that professional profiles are built without manual research.

AC: After import, NPI enrichment runs in batches (200/day). Matches by name + credential + enumeration year. Score threshold ≥ 5. Populates: npi, specialty, practice_city, practice_state, license_status. Status changes from "pending-enrichment" to "enriched".

PB-053P1

As an admin, I want imported alumni enriched with PubMed publication data (papers, journals, citations), so that published researchers are identified and tagged automatically.

AC: PubMed scan runs weekly (100/batch). Searches by author name. Stores: PMID, title, journal, pub date. Adds "published-researcher" tag. Creates publication touchpoints. Creates "new publication" trigger for queue.

PB-054P1

As an admin, I want enrichment to run on a schedule (NPI daily, PubMed weekly) with cursor tracking, so that all alumni are progressively enriched without manual intervention.

AC: NPI: daily 2AM, 200 records/batch, 200ms rate limit. PubMed: weekly Sunday 3AM, 100 records/batch, 500ms rate limit. Cursor persists across runs. Crash-safe (WAL). Full cycle completes in ~35 days (NPI) / ~70 weeks (PubMed) for 7,000 records.

PB-055P1

As a coordinator, I want enrichment to detect changes (practice move, credential update, license change) and generate triggers for my queue, so that I'm notified when something important changes.

AC: After each NPI refresh, diff previous triples vs current. If practice_city changed → practice_move trigger. If specialty changed → credential_change trigger. If license_status changed → alert. Triggers appear in queue with priority.

PB-056P2

As an admin, I want to see enrichment status (records enriched, pending, errors, last run time), so that I can monitor the pipeline health.

AC: Health endpoint or admin view: total records, enriched count, pending count, error count, last NPI run, last PubMed run, cursor position, estimated completion date.

PB-057P2

As an admin, I want to deduplicate alumni on import, so that re-uploading a CSV doesn't create duplicate records.

AC: On import, check for existing record by name + class_year + campus. If match found, skip (don't overwrite). Report duplicates count in import summary.

PB-058P3

As an admin, I want to manually trigger an enrichment refresh for a single alumnus, so that I can update a specific record without waiting for the batch schedule.

AC: Button on profile (admin only): [🔄 Refresh NPI]. Runs NPI + PubMed for this one record immediately. Shows result (updated fields or "no changes").

Epic 8 — Multi-Tenant & Branding

6 stories
PB-059P1

As an admin, I want each institution to have its own isolated data store, so that no institution can see another's alumni data.

AC: Per-tenant directory with encrypted triple store. Tenant ID in every request. Data path: /tenants/{id}/alumni.store/. PSK derived per tenant. Zero cross-tenant data leakage.

PB-060P1

As an admin, I want each institution to see their own branding (name, colors, logo) across all pages, so that the platform feels like their own tool, not a generic product.

AC: Tenant config triples: name, primary color, accent color, logo URL, tagline. Browser fetches config on boot. CSS variables injected. Nav, login, app all reflect tenant brand. Zero hardcoded institution names or colors.

PB-061P1

As an admin, I want tenant routing based on domain, so that lakemedical.org shows Lake Medical data and westernu.alumni-engine.com shows WesternU data.

AC: Worker reads Host header → maps to tenant_id. All subsequent requests scoped to that tenant. Unknown domains → error page. Custom domain or subdomain both supported.

PB-062P2

As an admin, I want tenant data encrypted at rest with a derived key, so that a compromised disk doesn't expose alumni data.

AC: AES-256-GCM encryption via store.h. Per-tenant key derived: HMAC-SHA-256(master_key, tenant_id). Encrypted on disk. Decrypted only in memory during server runtime.

PB-063P2

As an admin, I want session tokens scoped to a tenant, so that a session from one institution can't access another's data.

AC: Session token includes tenant claim. Server validates tenant claim matches request tenant_id. Mismatch → 403. No cross-tenant session reuse.

PB-064P3

As an admin, I want a support email configured per tenant, so that error pages and footers show the right contact for each institution.

AC: Tenant config includes support_email. Shown in footer and error pages. Defaults to platform support if not set.
