~I: QR Code — JS Encoder Broken

lakemedical.org qr.js vs NOUS qr.c · 4 fatal bugs found

What It Is

Two QR code encoders exist in the ecosystem:

C:\metal\auth\qr.cC606Working

C:\lakemedical.org\alumni\js\utils\qr.jsJS413Broken — unscannable output

The C version (deployed on the prime) generates valid, scannable QR codes. The JS version (in the browser) was written as a port but has 4 critical bugs that make its output unreadable by any QR scanner.

What the C Version Does Right

  • GF(256) arithmetic — primitive poly 0x11D, log/exp tables, correct Reed-Solomon
  • Multi-block interleaving — proper data/EC block subdivision per ISO 18004
  • All 4 penalty rules — runs, 2×2 blocks, finder patterns, dark module balance
  • Correct format info placement — 15-bit format string, BCH error correction, correct bit positions
  • Full test suite — test_qr.c (492 lines, 200+ assertions, tests all versions + EC levels)

Bugs in the JS Version

FATAL 1: Format Info Bit Placement Reversed

The format information (15-bit sequence encoding EC level + mask pattern) is placed at specific positions around the finder patterns. The JS version reverses the bit extraction order and uses different row/column mappings than the standard.

// JS (WRONG) — bits extracted in reverse order
var bit = (bits >> (14 - i)) & 1;  // Should be (bits >> i) for first area
// Also: row mapping skips positions incorrectly

// C (CORRECT) — matches ISO 18004 Table C.1
static const int r1[] = {0,1,2,3,4,5,7,8,8,8,8,8,8,8,8};
static const int c1[] = {8,8,8,8,8,8,8,8,7,5,4,3,2,1,0};

Effect: Scanner cannot determine EC level or mask pattern. QR is completely undecodable.

FATAL 2: No Block Interleaving

QR versions 4+ require multiple data/EC blocks with interleaved codewords. The JS version treats all data as a single monolithic block.

// JS: linear — data[], then EC[] (WRONG for V4+)
// C:  interleaved — data[blk0][0], data[blk1][0], data[blk0][1], ...
//     then EC[blk0][0], EC[blk1][0], ...

Effect: Data codewords placed in wrong bit positions for V4–V10. Decoder extracts garbage.

FATAL 3: Only 1 of 4 Penalty Rules Implemented

Mask selection requires scoring all 8 mask patterns using 4 penalty rules. The JS version only implements Rule 1 (consecutive runs). Missing:

  • Rule 2: 2×2 same-color blocks (3 pts each)
  • Rule 3: Finder-like patterns 10111010000 / 00001011101 (40 pts each)
  • Rule 4: Dark module proportion deviation from 50% (10 pts per 5%)

Effect: Wrong mask selected. Even if other bugs were fixed, the QR would be noisy and hard to scan.

BUG 4: Capacity Table Mismatch for Multi-Block Versions

The JS capacity array is simplified and diverges from the C version’s per-EC-level tables at V4+. Combined with the interleaving bug, the data payload length can be wrong.

The Fix: Port C to JS Faithfully

The C version (qr.c) is proven, tested (200+ assertions), and deployed in production on the prime server. The fix is straightforward:

  1. Port qr.c to JavaScript line-by-line, preserving all algorithms
  2. Include multi-block interleaving, all 4 penalty rules, correct format info placement
  3. Keep the same API: qr_encode(text) → {data, size}
  4. Test with known otpauth URIs against a scanner

Open Questions

  • Should the JS port support V7–V10? The C version tops at V6 (41×41). Most otpauth URIs fit in V4–V5.
  • Should we add version info placement for V7+? (Not needed if we cap at V6)
  • The prime already generates QR data server-side. Could we just send the pre-rendered matrix instead of re-encoding in the browser? (Simpler but adds wire payload.)
