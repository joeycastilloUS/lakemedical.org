Plan: Dual-Write — store.h Triples + Firestore Mirror

store.h is the source of truth · Firestore is the cloud-readable mirror · All data is triples

PLAN — AWAITING APPROVAL

Intent

Use store.h triples as the single canonical data model for everything — alumni records, enrichment data, signals, config, sessions, audit trail. Every piece of state is a triple: {subject, predicate, object}.

Simultaneously mirror all triple writes to Google Firestore as a cloud-accessible read replica. The C triple store on our metal is authoritative. Firestore is a convenience layer for:

  • Browser reads without custom API endpoints
  • Real-time subscriptions (Firestore SDK onSnapshot)
  • Admin dashboards and analytics queries
  • Disaster recovery (data lives in two places)
  • Third-party integrations that speak Firestore natively

The traps to avoid

RiskMitigation
Dual-write consistencyWrite store.h first, then Firestore. If Firestore fails, queue and retry. store.h is always correct. Firestore is eventually consistent. Never read from Firestore to make server-side decisions.
Cost at scaleFirestore charges per write ($0.18/100k). A 7,000-alumni NPI refresh = 7,000+ triple updates = ~35,000 doc writes/day. Batch writes (500 ops/batch) reduce cost. Project to document (not triple-per-doc) reduces writes 10x.
Data model mismatchTriples are S/P/O. Firestore is documents/collections. Don't store raw triples in Firestore — project them into documents. See mapping below.
Encryption gapstore.h uses app-level AES-256-GCM. Firestore uses Google-managed encryption (TLS in transit, AES-256 at rest via Google). Accept this: Firestore is not your vault. Don't put secrets (TOTP keys, PSKs) in Firestore. Only alumni data + signals.
Latency on writesFirestore write is async. alumni-server writes to store.h (local, <1ms), then fires Firestore write in background thread. User doesn't wait.

What does NOT go to Firestore

  • TOTP secrets — stay in vault.h only. Never leave the VM.
  • Session tokens — stay in store.h. Firestore is not a session store.
  • Encryption keys / PSKs — stay in vault.h.
  • Factory rule definitions — these ship with the binary. No need to mirror.
  • Audit/trace logs — too high volume. Stay in trace.h/pulse.h.

Rule: if it's a secret or a system internal, it stays on our metal. If it's data a browser user should see, it goes to Firestore.

Key change from previous plan

The browser no longer reads data through Workers. It reads alumni data, signals, and dashboard data directly from Firestore via the Firebase JS SDK. Workers only handle auth (wire protocol). This eliminates:

  • The /api/data Worker endpoint
  • The /api/refresh Worker endpoint
  • The /api/seed Worker endpoint
  • KV namespace for alumni data cache

These are replaced by Firestore reads with real-time subscriptions. Simpler. Faster. Less Worker code.

Triple-to-Firestore Data Model Mapping

The projection: triples → documents

Raw triples in store.h are {subject, predicate, object}. Storing each triple as a Firestore document is wasteful (too many docs, too many reads). Instead, project by subject: all triples sharing a subject become one Firestore document.

store.h triples (source of truth):
  alumni:jsmith   name            "Dr. Jane Smith"
  alumni:jsmith   class-year      "2018"
  alumni:jsmith   campus          "pomona"
  alumni:jsmith   specialty       "family-medicine"
  alumni:jsmith   practice-city   "Orange"
  alumni:jsmith   practice-state  "CA"
  alumni:jsmith   has-npi         "1234567890"
  alumni:jsmith   enriched-at     "2026-04-02T02:15:00Z"
  alumni:jsmith   status          "enriched"

        ↓ projection (group by subject)

Firestore document:
  Collection: tenants/{tenant_id}/alumni
  Document ID: jsmith
  Fields: {
    "name":           "Dr. Jane Smith",
    "class-year":     "2018",
    "campus":         "pomona",
    "specialty":      "family-medicine",
    "practice-city":  "Orange",
    "practice-state": "CA",
    "has-npi":        "1234567890",
    "enriched-at":    "2026-04-02T02:15:00Z",
    "status":         "enriched",
    "_updated":       Timestamp (server)
  }

One alumni = one document. Predicates become field names. Objects become field values. Adding a new predicate in store.h = adding a new field in Firestore. No schema changes.

The Dual-Write Pipeline

Schematic: Write path

Application code (CSV import, NPI refresh, trigger generation):
  │
  ├─ 1. triple_store_insert(store, "alumni:jsmith", "specialty", "family-medicine")
  │     → store.h writes to local encrypted mmap file
  │     → journal.h logs to WAL
  │     → DONE. Sub-millisecond. Authoritative.
  │
  ├─ 2. mirror_queue_push(subject, predicate, object, OP_INSERT)
  │     → Pushes to in-memory ring buffer (lock-free)
  │     → Returns immediately. Non-blocking.
  │
  └─ 3. Mirror thread (background):
        │
        ├─ Reads from ring buffer
        ├─ Groups by subject (batches predicates for same subject)
        ├─ Projects triples → Firestore document fields
        │
        ├─ HTTP POST to Firestore REST API:
        │    POST https://firestore.googleapis.com/v1/
        │      projects/{project}/databases/(default)/documents/
        │      tenants/{tenant}/alumni/{id}
        │    Body: { fields: { "specialty": { "stringValue": "family-medicine" } } }
        │    updateMask: { fieldPaths: ["specialty"] }
        │
        ├─ On success: advance cursor, clear from buffer
        ├─ On failure: retry with exponential backoff (1s, 2s, 4s, max 60s)
        └─ On persistent failure: log to trace.h, skip after 5 retries
           (store.h remains correct; Firestore catches up on next write)

Schematic: Delete path

triple_store_delete(store, "alumni:jsmith", "practice-city", NULL)
  │
  ├─ 1. store.h deletes matching triples
  │
  ├─ 2. mirror_queue_push("alumni:jsmith", "practice-city", NULL, OP_DELETE)
  │
  └─ 3. Mirror thread:
        ├─ Firestore updateDoc with FieldValue.delete() on "practice-city"
        └─ If ALL predicates for a subject are deleted → deleteDoc

Schematic: Batch enrichment (NPI refresh)

NPI enrichment batch (200 alumni):
  │
  for each alumnus:
  │  ├─ triple_store_insert() × 6-8 predicates   (local, fast)
  │  ├─ mirror_queue_push() × 6-8 predicates     (buffered)
  │  └─ sleep(200ms)                              (NPI rate limit)
  │
  After batch completes:
  │
  Mirror thread flushes buffer:
  ├─ Groups 200 alumni × ~7 predicates = ~1400 triple ops
  ├─ Projects to 200 Firestore document updates
  ├─ Sends as Firestore batch writes (500 ops/batch)
  │    → 1 batch commit (200 doc updates)
  └─ Total Firestore cost: 200 write ops = $0.00036
     (7,000 alumni / 200 per day = 35 days = $0.013 total)

The Mirror Module (firestore_mirror.c)

Pure C Firestore client — no SDK, just HTTP

Firestore has a REST API. We don't need the Firebase Admin SDK (Node.js). We use http.h to POST JSON directly.

firestore_mirror.c
  #include "http.h"        // HTTPS POST to Firestore REST API
  #include "crypt.h"       // SHA-256 for JWT signing (service account auth)
  #include "read.h"        // JSON building for Firestore document format

API:
  bool mirror_init(const char* project_id, const char* service_account_json);
  void mirror_close(void);

  // Queue operations (called from main thread, non-blocking)
  bool mirror_queue_insert(const char* tenant, const char* subject,
                           const char* predicate, const char* object);
  bool mirror_queue_delete(const char* tenant, const char* subject,
                           const char* predicate);

  // Background flush (called from mirror thread)
  int  mirror_flush(void);          // flush pending → Firestore, returns count
  int  mirror_pending(void);        // items in queue
  bool mirror_healthy(void);        // Firestore reachable?

  // Full re-sync (disaster recovery)
  int  mirror_resync(TripleStore* store, const char* tenant);
                                    // re-project entire store → Firestore

Service account auth (Pure C JWT)

Firestore REST API requires OAuth2 token.
Service account JSON → extract private key → sign JWT → exchange for access token.

1. Build JWT header: {"alg":"RS256","typ":"JWT"}
2. Build JWT claims:
     iss: service_account_email
     scope: "https://www.googleapis.com/auth/datastore"
     aud: "https://oauth2.googleapis.com/token"
     iat: now
     exp: now + 3600
3. Sign with RSA private key (crypt.h)
4. POST https://oauth2.googleapis.com/token
     grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer
     assertion={jwt}
5. Cache access_token for 1 hour, refresh before expiry

All in C. No SDK. No npm. Just http.h + crypt.h.

Firestore security rules (tenant isolation)

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Tenant data — read requires valid session token in custom claim
    match /tenants/{tenantId}/{collection}/{docId} {
      allow read: if request.auth != null
                  && request.auth.token.tenant == tenantId;
      allow write: if false;  // NEVER. Only alumni-server writes.
    }
  }
}

Browser reads require Firebase Auth token with tenant claim. Writes are server-only via service account. Users can never modify data through the browser.

Consistency Model

store.h = strong consistency (local)

All server-side reads come from store.h. Enrichment, triggers, auth decisions, admin queries — all read from the local triple store. Zero network hops. Sub-millisecond.

Firestore = eventual consistency (mirror)

Browser reads come from Firestore. After a write to store.h, the mirror thread pushes to Firestore. Typical latency: 50–200ms. Acceptable for a dashboard.

Conflict resolution: store.h always wins

1. Browser NEVER writes to Firestore (security rules: write = false)
2. Only alumni-server writes to Firestore (via service account)
3. If Firestore is stale → mirror_resync() re-projects entire store
4. If Firestore has unexpected data → overwrite from store.h
5. No bidirectional sync. No merge conflicts. One direction only.

Constraints

  • store.h is always authoritative. Firestore is a projection, never a source. Server-side code never reads from Firestore.
  • No Firebase Admin SDK. Pure C talks to Firestore REST API via http.h. No Node.js on the server.
  • Secrets never leave the VM. TOTP, PSKs, session tokens, encryption keys — store.h and vault.h only.
  • One-way mirror. store.h → Firestore. Never Firestore → store.h. No bidirectional sync.
  • Browser writes are forbidden. Firestore security rules enforce write: false. Only the service account writes.
  • Async, non-blocking. Mirror writes never block the main thread. If Firestore is down, store.h keeps working.
