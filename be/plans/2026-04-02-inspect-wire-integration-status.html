lakemedical.org ↔ NOUS relay — what's built, what's working, what's next

What It Is

TOTP-authenticated alumni tracking via NOUS wire protocol. Browser packs binary NTRP triples, Worker encrypts into 0xCA envelope, TCP to relay, relay dispatches (auth/register/forward), response comes back the same path in reverse.

E2E Test Results (20/20)

The E2E test proves the chain from browser to relay connects but returns 502:

  • ✓ Worker receives NTRP, validates magic, derives cockpit key
  • ✓ Worker encrypts + TCP connects to relay.3-nous.net:8080
  • ✓ Relay receives 0xCA envelope, decrypts with cockpit key
  • ⚠ Relay dispatches auth → relay_wire_to_prime() → prime response comes back but Worker can't decrypt the response envelope (502)

The 502 means: the relay-to-prime leg works (relay sends wire auth to prime), but the response coming back isn't a valid 0xCA envelope that the Worker can decrypt. This is the gap — the relay encrypts the response with its own key, not the cockpit key the Worker expects.

Open Questions

  1. Response encryption key mismatch? The relay encrypts responses with g_cockpit_key (line 1710). The Worker decrypts with the same derived cockpit key. Both derive from the same PSK + label. If the PSK matches on both sides, this should work. Does the relay have the same PSK as the Worker?
  2. Is a prime actually running and reachable? The relay has CLI primes (35.236.118.2:8080, 34.81.178.230:8080) from systemd config. Are those primes actually running and accepting wire auth?
  3. Prime auth handler: When the relay sends wire auth triples to a prime, does the prime have a handler that validates TOTP and returns a success/fail response in wire format?
  4. TOTP user provisioning: No users are registered on the prime. Even if auth gets through, there's no user to authenticate. Need at least one test user with a TOTP secret.

What's Next

The browser-to-relay path is fully built. The gap is the relay-to-prime leg and having a real auth round-trip work end-to-end. To close the loop:

  1. Verify prime is running and accepts wire auth requests
  2. Provision a test user with TOTP secret on the prime
  3. Debug the 502 — is it a key mismatch, a prime rejection, or a response format issue?
  4. Get a full auth round-trip: browser login → session token → gate opens
